# 🎓 《大模型营销平台》前置研发档案（阶段一 至 阶段五）

> 这是一份为了毕业设计答辩及论文撰写而专门归档的核心架构演进图谱。记录了本系统从零开始成长至今的每一次技术跨越与痛点解决。

---

## 🚀 阶段一：破冰与框架基建 (MVC与底层通信)
- **痛点**：缺乏一个现代化的 AI 对话业务系统基座。
- **技术实现**：
  - 采用 **Spring Boot + MyBatis-Plus** 搭建了经典的基于 REST API 的后端 MVC 分层治理结构（Controller, Service, Entity）。
  - **核心组件**：手写封装了底层网络骨架 `LLMUtil`。利用 `OkHttp3` 连接池实现了 OpenAI 格式的标准化 HTTP 对接协议，成功打通了 DeepSeek 接口，完成了项目从 0 到 1 的“能说话”。

## 🔌 阶段二：配置解耦与多模型弹性路由
- **痛点**：初版将请求地址和 Key 硬编码在 Java 类中，如果要接其他大模型极其繁琐且需要重启项目重新编译。
- **技术实现**：
  - **动态映射架构**：在 `application.yml` 中建立了 `llm.providers` 的配置树。
  - **动态工厂**：使用 Spring 的 `@ConfigurationProperties` 自动将其装载为内存中的 `Map<String, ModelConfig>`。
  - **前端联动**：让前端通过下拉菜单传入 `modelKey`。后端直接通过 Key 从 Map 中 `O(1)` 时间复杂度极其优雅地拿到底层参数，实现了千问、豆包、智谱等数十个大模型的**无痛热插拔**接入。

## 🔐 阶段三：JWT 全链路安全鉴权体系
- **痛点**：AI API 是要花自己的钱调用（或者是花额度）的，绝对不允许任何人顺便刷接口，必须有账号体系。
- **技术实现**：
  - **加密存储**：搭建了 `sys_user` 实体，用户的账号和登录密码全系使用 `BCrypt` 散列算法加盐处理，确保落盘数据脱敏。
  - **无状态鉴权**：采用 `Hutool-JWT` 生成前端授权令牌。
  - **切面拦截**：编写了 `LoginInterceptor`。任何一次发消息的请求都会在被 Controller 收到之前，经过拦截器进行 Token 解析。解析通过后，将用户 ID 放入 `ThreadLocal`（线程安全缓存 `UserContext`）供全域无感调用。

## 🎨 阶段四：前端视效重构与 Markdown AST 渲染
- **痛点**：浏览器默认提示丑陋，且大模型吐出的代码和层级内容粘成一团，严重影响了作为“产品级应用”的观感。
- **技术实现**：
  - **全局 Toast 组件**：写了一套带有进入、退出动画效果（Animation Keyframes）的悬浮提示框，彻底淘汰原始 `alert()`。
  - **高级解析与防注入**：引入了 `marked.js` 将文本渲染为带有标题和独立代码块的富文本，并保留了基础防止 XSS 攻击的转义原则。
  - **Ins 极简风范**：以毛玻璃（Glassmorphism）和软阴影（Soft Shadows）作为视觉基座，将生硬的四角边框打磨为高度统一的药丸/胶囊级圆角排版，将系统质感拉伸到了硅谷初创企业级别。

## 🧠 阶段五：20轮长尾滑动跨模态记忆体（高光架构）
- **痛点**：“一次性聊天”毫无上下文体验，传统的 RAG 技术不适用于聊天记忆，而把所有历史记录放在前端则容易被人篡改。
- **技术实现（Data-Driven 理念的核心应用）**：
  - **MySQL 记忆落盘**：基于 `chat_history` 表将用户的每一次发言以及 AI 的每一次回复按照时间戳固化。
  - **滑动窗口提纯**：在接收到新消息瞬间，拦截该 `SessionId` 下最近的 20 轮历史，并通过 `FastJSON` 瞬间重组为多节点数组。
  - **深层洗脑与修正 (Prompt Engineering)**：由于跨模型热切换会导致身份血统污染（比如智谱扮演成了千问），我在底层组装时强制注入了一条最高权限的 `role: system` 指令：“无论前文说了什么，请认识到你现在的身份是...”，通过系统级洗脑，完美实现了**不同公司的大模型能够在一个对话框里跨模型继承彼此的记忆，且各司其职**。
